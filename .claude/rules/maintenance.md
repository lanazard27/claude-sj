# 메모리 관리, 하네스 유지보수, 피드백 규칙

## "메모리 정리해" 프로시저

**트리거**: 정확히 "메모리 정리해"만 실행. "최적화해", "정리해", "압축해"는 `rules/Optimization.md` 담당.

### 실행 단계
1. **진단** — MEMORY.md 줄 수(150줄+ 시 압축) / gc.md 항목 수 / reflexion-patterns.jsonl 항목 수·오래된 항목 / sessions/ 스냅샷(7일+)
2. **정리** — MEMORY.md: 150줄+ 시 최근 120줄 유지 / gc.md: 살아있으면 카테고리 이관, 30일+ 미사용 삭제 / reflexion: 30일+ resolved 삭제, 동일 패턴 병합, 50개+ 시 오래된 것 삭제 / sessions: 7일+ 스냅샷 정리 / experience: 동일 카테고리 5개+ 시 인사이트로 일반화→개별 삭제
3. **보고** — 전후 비교(줄 수/항목 수), 삭제/이관/압축 항목, 남은 항목 수

### 기본 원칙
- 프로젝트 메모리는 별도 디렉토리 관리 — 홈 MEMORY.md에 프로젝트 내용 금지
- 홈 MEMORY.md는 사용자 선호/규칙만 / 프로젝트별 내용은 해당 프로젝트 memory에
- 분야별 선호 패턴, 스타일 → MEMORY.md / 실패 이력, 반복 수정 → 기록

### 자동 최적화 (세션 종료 시)
- MEMORY.md: 150줄+ 시 최근 120줄 유지 / 스냅샷: 7일+ 자동 정리 / experience: 5개+ 시 압축 권고 / 통계: 30일+ 자동 정리

### 수동 관리 (월 1회)
CLAUDE.md·settings.json·memory에서 무효 규칙 정리 / agents/ 점검(위임 적절성, 신뢰도 임계값, 도구 권한, 사용 빈도) / 반복 무시 규칙 분석→표현 수정 / 새 작업 패턴→규칙화→CLAUDE.md 반영 / permissions 갱신 / experience 압축(동일 카테고리 5개+→인사이트로 일반화) / GC 수거(gc.md: 살아있으면 이관, 30일+ 미사용 삭제)

## 피드백 규칙
실수 발견 시: 원인 분석+수정안 → **사용자 확인** → 승인 시 memory 기록 / 기록된 수정안은 이후 프롬프트에 자동 포함("이건 절대 하지 마") / 거부 시 폐기 또는 피드백 재수정 / 기존 수정안도 주기적 재검증 — 더 나은 방법 있으면 업데이트 제안

## Reflexion Memory
에러 패턴 구조화 기록, 다음 세션에서 재발 방지.
**저장**: `.claude/memory/reflexion-patterns.jsonl`
**형식**: `{"date":"YYYY-MM-DD","pattern":"에러 패턴명","cause":"근본 원인","fix":"해결책","context":"프로젝트/상황"}`
**기록 트리거**: 같은 에러 2회 연속 / 빌드 실패 후 원인 파악 / 코드 리뷰 반복 지적 / 사용자 "이거 전에도" 언급
**활용**: 세션 시작 시 읽기 / 에이전트 스폰 시 관련 패턴이면 "이 패턴 피해라" 포함 / 월 1회: 30일+ 재발 없으면 삭제, 동일 패턴 병합
**유지**: 최대 50개(초과 시 오래된 것 삭제) / 재발 시 `last_recurred` 업데이트 / 3회 연속 재발 없으면 `resolved: true`
