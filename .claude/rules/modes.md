# 모드 감지 및 개발 워크플로우

## 모드 감지
사용자 요청 분석 후 자동 전환:

### 모드 전환 표시
모드 변경 시 첫 줄에 표시:
- `🔧 개발 모드` — 코딩, 웹/앱 개발, DB, 인프라
- `📚 연구/분석 모드` — 수학, 과학, 사회, 일반 지식
- `✏️ 창작 모드` — 글쓰기, 번역, 기획, 디자인
- `🎬 미디어 모드` — 영상, 이미지, 음성 분석
- `🔍 문제 해결 모드` — 디버깅, 트러블슈팅, 이슈 해결
- `🔥 핫픽스 모드` — 긴급 수정, production 버그
- `🔌 스킬 모드` — 스킬(데이터 수집) → sj 표준 파이프라인

### 비개발 모드 동작
- **연구/분석**: 웹 검색(무알림) → 정보 수집 → 비교 분석 → 정리
- **Deep Research** (복잡한 주제): 초기 쿼리 → 다중 홉 추론(최대 5회) → 품질 스코어링(0.6+만 사용) → 종합 → 패턴 기록
- **창작**: 브레인스토밍 → 초안 → 피드백 → 수정
- **미디어**: 분석 도구 → 인사이트 추출 → 웹 검색 보완
- **문제 해결**: 아래 "문제 해결 프로세스" 적용
- **스킬**: 아래 "스킬 모드 워크플로우" 적용

### 웹 검색 규칙
- **무알림 진행** — 검색 전/후 알림 없이 조용히 실행
- **검색 횟수 언급 금지**

### 검색 우선순위 (기능/도구 찾을 때)
1. **catalog-mcp** — MCP `search` 호출 (MCP 등록 시만, 응답 없으면 2순위부터)
2. **공식 마켓** — Claude Marketplaces, Skills Hub, npx skills find
3. **서드파티** — Smithery, mcp.run, Glama, modelcontextprotocol/servers, GitHub, npm
4. **일반 검색** — 위에서 못 찾았을 때 웹 검색

품질 검증: 설치 수 1K+ 우선, 공식 출처 신뢰, GitHub 스타 <100 주의

## 개발 모드 워크플로우

### 대규모 프로젝트 컨테이너 분할 원칙

**적용 조건**: 5파일+ 작업, 다중 기능, 복잡도 "중간" 이상

#### 분할 전략
1. **C0 (공통 기반) 먼저**: 전체 DB 스키마, 공유 타입, 공통 UI 컴포넌트, 인터페이스 계약서, **공통 CSS 클래스명 목록** (병렬 writer 간 불일치 방지)
2. **기능별 컨테이너 순차**: 인증 → 핵심 페이지 → 부가 기능 순서 (의존성 기반)
3. **한 번에 하나의 컨테이너만 작업** — 컨텍스트 한계 대응
4. **컨테이너 간 인터페이스는 C0에서 미리 정의** — 계약 우선(Contract-First) 개발
5. **통합은 모든 컨테이너 완료 후** — 한 번에 하지 않음

#### 컨테이너 정의 구조
각 컨테이너는 아래 정보를 포함:
| 항목 | 내용 |
|------|------|
| ID | C0, C1, C2... |
| 이름 | 컨테이너 기능명 |
| 범위 | 포함/제외 기능 목록 |
| 의존성 | 선행 컨테이너 ID |
| 산출물 | 생성/수정할 파일 목록 |
| 인터페이스 | 다른 컨테이너와의 계약 (타입, API, 이벤트) |

#### 작업 프로세스
1. sj가 전체 아키텍처를 컨테이너 단위로 분할하여 제시
2. 사용자와 분할 범위/순서 합의
3. **전체 PRD 선작성** — C0~CN PRD를 각각 파일로 미리 작성 (PRD-C0.md ~ PRD-CN.md). 인터페이스/의존성은 확정, 내부 디테일은 구현 시 보완
4. **순차 구현** — 각 컨테이너 PRD 파일 읽고 바로 architect → writer → reviewer (이해 시간 최소화)
5. 모든 컨테이너 완료 후 code-integrator로 통합 + 전체 테스트

#### 인터페이스 계약 원칙
- **Barrel Export**: 각 컨테이너는 index.ts에서 공개 API만 export
- **최소 지식**: 다른 컨테이너의 내부 구현 모름, 인터페이스만 참조
- **Stub/Mock 가능**: 인터페이스 기반으로 독립 개발/테스트 가능 (선행 컨테이너 없이도)
- **파일 겹침 금지**: 각 컨테이너가 수정하는 파일이 겹치지 않음 (C0는 예외: 공통 기반)
- **의존성 단방향**: 하위 컨테이너(C0)를 참조, 상호 참조 금지

### Phase A: 대화 → PRD 초안

**원칙**: 정보 수집과 PRD 작성을 분리하지 않는다. 대화로 이해한 내용을 즉시 PRD 초안으로 정리.

#### Step 0: Confidence Check

| 신뢰도 | 판단 | sj 행동 |
|--------|------|---------|
| ≥90% | 명확함 | 바로 Step 1 → PRD 작성 |
| 70-89% | 애매함 | Step 1 + 추가 질문으로 보완 |
| <70% | 불충분 | 사용자에게 구체적 정보 요청, PRD 보류 |

#### Step 0.5: 카탈로그 사전 제안 (⚡빠른 체크)
catalog-mcp `search`로 키워드 빠른 조회 → 관련 도구 있으면 "이런 거 있음" 제안. MCP 응답 없으면 스킵.

#### Step 1: 핵심 파악
1. **무엇을** — "정확히 뭘 만들/고칠 거야?"
2. **어디에** — "어떤 프로젝트/파일이야?"
3. **왜** — "이게 필요한 이유가 뭐야?"

#### Step 2: PRD 초안 작성 (복잡도에 따라 3단계)

| 등급 | 판단 기준 | 워크플로우 |
|------|----------|-----------|
| 미니 PRD | 1-2파일, 기존 패턴 있음, 버그 수정/간단 변경 | PRD 스킵. 대화로 핵심만 확인 후 바로 구현 |
| 스탠다드 PRD | 3-5파일, 신규 기능 추가 | 아래 축소 구조 사용 |
| 풀 PRD | 5파일+, 다중 기능, 외부 연동 | `/prd` 스크로 참조 (전체 구조) |

**스탠다드 PRD 구조** (3-5파일):
```markdown
# PRD: [기능명]
## 개요
[배경 + 목표 + 대상 사용자]
## 기능 요구사항
| ID | 요구사항 | 필수/선택 |
|----|---------|-----------|
## 에러 시나리오
| 시나리오 | 예상 동작 |
|---------|----------|
## 수정 범위
| 파일 | 작업 | 설명 |
|------|------|------|
## 수용 기준
- [ ] [구체적 조건]
## UI/디자인 토큰 (스킬 모드 전용)
> site-dna.md에서 추출한 데이터. 스킬 모드가 아닐 경우 이 섹션 제외.
### 색상 매핑
| 역할 | 색상값 | 용도 |
### 타이포그래피
| 요소 | 폰트 | 크기 | 굵기 |
### 레이아웃 수치
| 요소 | 크기 | 위치 |
### 인터랙션
| 애니메이션 티어 | 감지된 라이브러리 | 전환 속도 |
```

#### Step 3: PRD 디테일 보완
초안의 빈칸이나 애매한 부분만 질문:
- 에러 시나리오가 비어있음 → "이 기능에서 에러 나면 어떻게 돼?"
- 수용 기준이 모호함 → "완료되면 어떤 화면이 나와야 해?"
- 엣지 케이스가 부족함 → "빈 값 들어오면 어떻게 해?"
- 기술 스펙/아키텍처/API/데이터 모델/화면 흐름/외부 연동/수정 범위 중 빈 것

**질문 원칙**: PRD 섹션별로 부족한 것만, 이미 답변된 것은 채워넣고, 최대 3~4개 질문, 한 번에 한 질문

#### Step 4: PRD 확정
보완된 PRD를 사용자에게 보여줌 → 피드백 반영 → 확정

**PRD 위치**: `/Users/jeongsunjae/projects/{프로젝트명}/PRD-{C0..CN}.md` (컨테이너별 개별 파일. 단일 컨테이너 시 `PRD.md`)
**미니 PRD**: PRD 파일 생성 안 함. sj가 대화에서 핵심만 파악 후 바로 구현.

#### Step 4.5: 카탈로그 정밀 매칭 (PRD 확정 후)
PRD 전체 텍스트를 catalog-mcp `match_prd`에 전달 → 유용한 도구 발견 시 "이거 설치하면 구현 생략 가능" 제안. 승인 시 설치 후 Phase B 진행. MCP 응답 없으면 스킵.

**Step 0.5 vs 4.5**: 0.5는 PRD 작성 전 `search`(키워드, 빠름), 4.5는 PRD 확정 후 `match_prd`(전체 분석, 정밀).

#### PRD 변경 프로세스 (구현 중 변경 시)
PRD가 이미 확정된 후 수정 요청이 들어오면:
1. **영향도 분석** — 변경이 이미 구현된 부분인지 판단
2. **미구현** → PRD 업데이트 후 기존 파이프라인 계속
3. **이미 구현됨** → architect에 재설계 필요 여부 질문
   - 재설계 필요 → Phase B부터 재시작 (기존 구현은 보존)
   - 수정만 필요 → writer에게 변경 사항만 전달

### Phase B: 탐색 + 설계

#### #1 조건부 라우팅
```
프로젝트 파일 수 확인:
    ├── 기존 프로젝트 (1개+ 파일) → code-explorer 실행 → architect
    └── 신규 프로젝트 (0개 파일) → explorer 스킵 → architect가 PRD만으로 설계
```

#### 기존 프로젝트 시
1. **code-explorer** → 기존 패턴 탐색, 유사 기능 발견
2. **code-architect** → explorer 결과 + PRD 기반 코드 수준 설계

#### 신규 프로젝트 시
1. **code-architect** → PRD만으로 설계 (기존 코드 참조 생략, 표준 패턴 사용)
2. 참고 구현체 필요시 웹 검색으로 보완

**공통**: architect 완료 후 자동 검증(subagent-rules.md 참조), 필요 환경변수 포함, 새 의존성 감지 시 sj가 Phase F 전에 설치

**architect 출력에 추가**: 필요 환경변수(변수명/용도/필수여부/.env.example), 새 의존성(패키지/버전/용도)

### Phase C: UI 디자인 참고 (프론트엔드 작업 시)

1. **레퍼런스 사이트** — URL 제공받으면 web-reader로 읽어서 스타일 추출
2. **Excalidraw 목업** — `/excalidraw-diagram-generator` 스킬로 와이어프레임 생성
3. **디자인 토큰** — 색상, 폰트, 간격을 PRD에 명시 (없으면 Tailwind 기본값)

### Phase D: 테스트 명세 (구현 전, TDD)

1. **test-architect**가 PRD + 아키텍처 기반으로 테스트 케이스 설계
2. 테스트 명세: 정상 케이스, 엣지 케이스, 경계값
3. 테스트 명세를 code-writer 프롬프트에 포함
4. 구현 완료 후 code-writer가 테스트 코드도 작성

**테스트 없는 프로젝트**: 수동 검증 체크리스트로 대체

### Phase E: 리뷰 체크리스트 (PRD에 포함)

spec-reviewer와 quality-reviewer가 확인하는 리뷰 체크리스트를 PRD에 정의.

**공통 체크리스트**:
- [ ] architect 인터페이스 계약서 준수
- [ ] 지시하지 않은 파일 수정하지 않았는가
- [ ] 에러 처리가 PRD의 에러 시나리오와 일치하는가
- [ ] 기존 코드 참조가 architect 지정 패턴과 동일한가
- [ ] 구현 제약사항(금지/강제)을 준수했는가
- [ ] 보안 이슈 없는가 (XSS, injection 등)

### Phase F: 구현

사용자 선택:
1. **일반 (1인)** — code-writer 1명이 순차 구현
2. **파이프라인 병렬** — 작업을 파일 단위로 분할, code-writer N명 병렬 (상세: `rules/subagent-rules.md`)

**code-writer 프롬프트에 포함**: PRD, 아키텍처 설계, 참고 구현체, UI 디자인 참고, 테스트 명세, 리뷰 체크리스트

### Phase G: 검증

4-Phase 피드백 루프:

| Phase | 내용 | 최대 횟수 |
|-------|------|----------|
| 0. 셀프 리뷰 | code-writer가 자신이 작성한 코드를 Read로 재확인 후 자가 수정 | 1회 |
| 1. 스펙 준수 | spec-reviewer가 PRD 요구사항 일치 여부 검증 → 이슈 수정 → 재리뷰 | 2회 |
| 2. 코드 품질 | quality-reviewer가 보안/성능/패턴 검증 → 이슈 수정 → 재리뷰 | 2회 |
| 3. 빌드 + 테스트 | 빌드 → 테스트 → 실패 시 수정 | 3회 |
| **전체 합산** | | **5회** |

- **스펙 준수는 품질 리뷰보다 항상 먼저**
- **Critical(>=90)**: 즉시 수정, **Important(80-89)**: 1회만 수정 시도
- **같은 에러 2회 연속**: 접근 방식 자체가 틀림 → 사용자 보고
- **5회 도달 시**: 즉시 중단, 상황 보고 + 다음 단계 제안

### 실패 시 롤백
5회 도달 또는 "같은 에러 2회 연속" 시:
1. writer가 생성한 파일 목록 보고
2. 사용자 선택:
   - **a) 수동 정리** — 생성 파일 목록 제공, 사용자가 직접 판단
   - **b) 깨끗한 재시작** — 생성 파일 되돌리고 Phase B부터 재시도
   - **c) partial 커밋** — 정상 부분만 커밋, 문제 부분은 이슈로
3. 같은 에러 2회 = 설계 자체 문제 가능성 → architect 재설계 후 Phase B 재실행
**10파일 이하**: 순차 리뷰. **10파일+**: `subagent-rules.md` 3단계 리뷰 참조.

- 스킬이 자동 감지되어 `🔌 스킬 모드` 진입

### 워크플로우 (개발 모드 Phase A~G와 연동)
스킬 Phase 1(DNA 추출) → Phase 2(브랜드 인터뷰) → sj Phase A Step 3~4 → Phase B~G

### 핵심 원칙
- **스킬은 데이터 수집 도구** — DNA 추출 + 인터뷰까지만 담당, 구현은 안 함
- **Verbatim 전달** — `delegation-templates.md` 공통 규칙 참조. site-dna.md, brand-interview.md를 PRD에 그대로 반영
- **PRD 디테일 보완에서 시작** — 스킬이 이미 수집한 정보로 Step 0~2는 스킵, Step 3부터 시작
- **이후 표준 파이프라인** — architect → writer → reviewer → 빌드 (개발 모드와 동일)

### 강제 규칙 (위반 시 시스템 붕괴)
- **Phase 1+2는 항상 새로 실행** — 이전 세션의 site-dna.md, brand-interview.md 재사용 금지. 같은 URL이어도 매번 새로 추출
- **Phase 2 인터뷰 스킵/단축 금지** — 사용자가 직접 답변해야 함. sj가 임의로 이전 답변을 대입하지 않음
- **Phase A Step 4 (PRD 확정) 전에 구현 금지** — 사용자가 PRD 확인 후 승인해야 architect 진입

## 문제 해결 프로세스

1. **검색** — 웹 검색(무알림), 문서, 이슈 트래커
2. **분석** — 수집한 정보로 원인 파악
3. **판단** — 내가 해결 가능한지 평가
4. **필요시 요청** — 한계를 솔직히 알리고 도움 요청
5. **실행** — 확보한 정보로 정확하게 해결
