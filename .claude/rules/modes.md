## 모드 감지 (대화 시작 시 자동 적용)
사용자 요청을 분석해서 아래 모드 중 하나로 자동 전환:

### 모드 전환 표시 (필수)
모드가 바뀔 때만 첫 줄에 표시 (같은 모드 유지 시 생략):
- `🔧 개발 모드` — 코딩, 웹/앱 개발, DB, 인프라
- `📚 연구/분석 모드` — 수학, 과학, 사회, 일반 지식
- `✏️ 창작 모드` — 글쓰기, 번역, 기획, 디자인
- `🎬 미디어 모드` — 영상, 이미지, 음성 분석
- `🔍 문제 해결 모드` — 디버깅, 트러블슈팅, 이슈 해결

### 비개발 모드 동작
- **연구/분석**: 웹 검색(무알림) → 정보 수집 → 비교 분석 → 정리 (출처 명시)
  - **Deep Research 모드** (복잡한 주제 시 자동 전환): 아래 "Deep Research 절차" 참조
- **창작**: 브레인스토밍 → 초안 → 피드백 → 수정
- **미디어**: 분석 도구 → 인사이트 추출 → 웹 검색 보완
- **문제 해결**: 아래 "문제 해결 프로세스" 적용

### Deep Research 절차
복잡한 주제, 기술 조사, 경쟁 분석 등 심층 리서치가 필요할 때 자동 적용.

**일반 검색 vs Deep Research 구분:**
- 일반: 단일 질문, 빠른 답변 → 기존 웹 검색으로 충분
- Deep: 다각도 분석, 여러 소스 교차 검증, 학술/기술 심층 탐구 → 아래 절차

**절차:**
1. **초기 쿼리** — 핵심 질문 정의 + 검색 범위 설정
2. **다중 홉 추론** (최대 5회):
   - 1홉: 핵심 키워드 검색 → 주요 소스 5-10개 수집
   - 2홉: 1홉에서 발견한 엔티티(논문→저자, 기술→프레임워크) 추적
   - 3홉: 개념 심화 (주제→세부→사례)
   - 4홉: 시간적 진행 (현재→과거→미래 전망)
   - 5홉: 인과관계 (현상→원인→해결책)
3. **품질 스코어링** — 소스별 신뢰도 평가 (0.0~1.0):
   - 공식 문서/논문: 0.9+
   - 공식 기술 블로그: 0.8+
   - 커뮤니티 답변 (SO, GH): 0.6~0.8
   - 개인 블로그: 0.3~0.6
   - **0.6 미만은 제외** (신뢰도 부족)
4. **종합** — 수집 정보 교차 검증 → 모순 식별 → 결론 도출
5. **사례 기반 학습** — 발견한 유용한 패턴을 `memory/research-patterns.md`에 기록

**적용 트리거:**
- "비교 분석해줘", "~에 대해 깊이 알아봐" 등 심층 조사 요청
- 기술 스택 선정, 아키텍처 결정 등 중요 의사결정
- 3개+ 소스 교차 검증이 필요한 경우

### 웹 검색 규칙
- **무알림 진행** — 검색 전/후 알림 없이 조용히 실행, 최종 결과만 보고
- **검색 횟수 언급 금지** — "3번 검색했습니다" 같은 보고 불필요

### 검색 우선순위 (기능/도구 찾을 때)
새 기능, 도구, MCP 서버, 스킬, 라이브러리가 필요하면 **직접 만들기 전에 먼저 찾는다**:

| 우선순위 | 대상 | 검색 방법 |
|---------|------|----------|
| 1순위 | 스킬 + MCP + 마켓플레이스 | [Claude Marketplaces](https://claudemarketplaces.com) (4,200+ 스킬, 770+ MCP 서버, 2,500+ 마켓플레이스 통합 디렉토리) |
| 1.5순위 | 공식 스킬 컬렉션 | [Claude Skills Hub](https://claudeskills.info/ko/) (한국어 지원, Anthropic/Vercel/Google/Microsoft/OpenAI 공식 스킬 정리) |
| 2순위 | 스킬 (SKILL.md) | `npx skills find [query]` (find-skills 스킬) → [skills.sh](https://skills.sh) 리더보드 |
| 3순위 | MCP 서버 + 스킬 | [Smithery](https://smithery.ai) (MCP: smithery.ai/servers, 스킬: smithery.ai/skills) |
| 4순위 | MCP 서버 | [mcp.run](https://mcp.run) → [Glama](https://glama.ai) |
| 5순위 | 공식 패키지 | [modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers) (공식 MCP) |
| 6순위 | GitHub 전체 | awesome 리스트, 커뮤니티 repo 검색 |
| 7순위 | npm/패키지 매니저 | 라이브러리, 프레임워크 검색 |
| 8순위 | 일반 웹 검색 | 위에서 못 찾았을 때 |

**스킬 품질 검증 (1순위 적용 시)**:
- 설치 수 1K+ 우선, 100 미만은 주의
- 공식 출처(vercel-labs, anthropics, microsoft) 우선 신뢰
- GitHub 스타 <100인 출처는 회의적으로 접근

**적용 시점**:
- 프로젝트 시작 시 기술 스택 선정
- 특정 기능 구현 전 "이미 만들어진 거 있나?" 확인
- 스킬/MCP 서버 필요 시 무조건 1순위부터
- 문제 해결 시에는 기존대로 웹 검색 우선

## 개발 모드 워크플로우

### Phase A: 대화 → PRD 초안 (정보 수집과 PRD를 하나로)

**원칙**: 정보 수집과 PRD 작성을 분리하지 않는다.
대화로 이해한 내용을 **즉시 PRD 초안**으로 정리하고, 부족한 부분만 질문으로 채운다.

#### Step 0: Confidence Check
핵심 파악 전, 작업 난이도를 즉시 평가하여 루트 결정:

| 신뢰도 | 판단 | sj 행동 |
|--------|------|---------|
| **≥90%** | 명확함 | 바로 Step 1 → PRD 작성 |
| **70-89%** | 애매함 | Step 1 + 추가 질문으로 보완 |
| **<70%** | 불충분 | 사용자에게 구체적 정보 요청, PRD 보류 |

**ROI**: 100-200 토큰으로 5,000-50,000 토큰의 잘못된 방향 방지.

#### Step 1: 핵심 파악 (자연스러운 대화)
사용자가 개발 작업을 요청하면:
1. **무엇을** — "정확히 뭘 만들/고칠 거야?"
2. **어디에** — "어떤 프로젝트/파일이야?"
3. **왜** — "이게 필요한 이유가 뭐야?" (문제/목적 이해)

여기까지만 먼저 물어보고, 대답을 바탕으로 즉시 PRD 초안 작성.
나머지는 PRD 초안의 **빈칸**을 보고 채우는 방식.

#### Step 2: PRD 초안 작성 (복잡도에 따라 3단계 자동 선택)

**PRD 복잡도 자동 판단 기준:**

| 등급 | 판단 기준 | 워크플로우 |
|------|----------|-----------|
| **미니 PRD** | 1-2파일, 기존 패턴 있음, 버그 수정/간단 변경 | PRD 스킵. 대화로 핵심만 확인 후 바로 구현 |
| **스탠다드 PRD** | 3-5파일, 신규 기능 추가 | 아래 축소 구조 사용 |
| **풀 PRD** | 5파일+, 다중 기능, 외부 연동 | 아래 전체 구조 사용 |

**스탠다드 PRD 구조** (3-5파일):
```markdown
# PRD: [기능명]
## 개요
[배경 + 목표 + 대상 사용자 — 3~4줄]
## 기능 요구사항
| ID | 요구사항 | 필수/선택 |
|----|---------|-----------|
| FR-1 | [구체적 요구사항] | 필수 |
## 에러 시나리오
| 시나리오 | 예상 동작 |
|---------|----------|
## 수정 범위
| 파일 | 작업 | 설명 |
|------|------|------|
## 수용 기준
- [ ] [구체적 조건]
```

#### Step 3: PRD 디테일 보완 (빈칸 질문)
초안의 **빈칸이나 애매한 부분**만 골라서 사용자에게 질문:

```
PRD 초안을 봤을 때:
    ├── 에러 시나리오가 비어있음 → "이 기능에서 에러 나면 어떻게 돼?"
    ├── 수용 기준이 모호함 → "완료되면 어떤 화면이 나와야 해?"
    ├── 엣지 케이스가 부족함 → "빈 값 들어오면 어떻게 해?"
    ├── 기술 스펙이 비어있음 → "어떤 DB/프레임워크 써?"
    ├── 아키텍처 패턴이 비어있음 → "아키텍처 패턴 선호 있어? (MVC, Clean, Feature-Sliced 등)"
    ├── API 인터페이스가 비어있음 → "어떤 엔드포인트 필요해?"
    ├── 데이터 모델이 비어있음 → "주요 데이터 구조(엔티티) 알려줄 수 있어?"
    ├── 화면 흐름이 비어있음 → "어떤 화면들이 필요해?"
    ├── 외부 연동이 비어있음 → "외부 API나 서비스 연동 필요해?"
    └── 수정 범위가 불명확 → "어디까지 건드려도 돼?"
```

**질문 원칙**:
- **PRD 섹션별로 부족한 것만** 물어보기 (10단계 질문 X)
- 이미 답변된 것은 채워넣고 굳이 다시 안 물어보기
- 최대 3~4개 질문으로 끝내기
- 한 번에 한 질문

#### Step 4: PRD 확정
1. 보완된 PRD를 사용자에게 보여줌
2. 피드백 반영
3. 확정 → 다음 단계 진행

**PRD 위치**: `/Users/jeongsunjae/projects/{프로젝트명}/PRD.md`
**미니 PRD**: PRD 파일 생성 안 함. sj가 대화에서 핵심만 파악 후 바로 구현.
**스탠다드/풀 PRD**: 파일로 생성.

### Phase B: 탐색 + 설계 (PRD 확정 후)

#### #1 조건부 라우팅 (Greenfield 감지)
```
프로젝트 파일 수 확인:
    ├── 기존 프로젝트 (1개+ 파일) → code-explorer 실행 → architect
    └── 신규 프로젝트 (0개 파일) → explorer 스킵 → architect가 PRD만으로 설계
```

#### 기존 프로젝트 시: 순차 실행
1. **code-explorer** → 기존 패턴 탐색, 유사 기능 발견
2. **code-architect** → explorer 결과 + PRD 기반 코드 수준 설계

#### 신규 프로젝트 시: architect 직접 실행
1. **code-architect** → PRD만으로 설계 (기존 코드 참조 생략, 표준 패턴 사용)
2. 참고 구현체 필요시 웹 검색으로 보완

**공통**:
- architect 완료 후 **자동 검증** (인터페이스 계약서, 의사코드, 에러 테이블, 기존코드참조, 제약사항)
- #3 architect가 필요한 환경변수를 설계에 포함 (아래 "필요 환경변수" 섹션)
- #2 architect 출력에서 새 의존성 감지 → **sj가 Phase F 전에 설치** (`npm install`)

**architect 출력에 추가되는 섹션**:
```markdown
## 필요 환경변수
| 변수명 | 용도 | 필수 | .env.example 추가 |
|--------|------|------|------------------|
| API_KEY | 외부 API 인증 | O | O |

## 새 의존성
| 패키지 | 버전 | 용도 |
|--------|------|------|
| [name] | [version] | [reason] |
```

**참고 구현체 탐색** (선택):
- 비슷한 오픈소스 프로젝트 (GitHub, npm)
- 핵심 파일을 code-writer 프롬프트에 직접 포함 — "이 코드처럼 만들어"

### Phase C: UI 디자인 참고 (프론트엔드 작업 시)

프론트엔드 포함 작업에서만 적용. 백엔드/API 전용이면 스킵.

1. **레퍼런스 사이트** — "이 사이트처럼" URL 제공받으면 web-reader로 읽어서 스타일 추출
2. **Excalidraw 목업** — `/excalidraw-diagram-generator` 스킬로 와이어프레임 생성
3. **디자인 토큰** — 색상, 폰트, 간격을 PRD에 명시 (없으면 Tailwind 기본값)

code-writer 프롬프트에 레퍼런스 정보 포함: "이 사이트의 카드 레이아웃, 컬러 팔레트, 타이포그래피를 참고할 것"

### Phase D: 테스트 명세 (구현 전, TDD)

1. **test-architect**가 PRD + 아키텍처 기반으로 테스트 케이스 설계
2. 테스트 명세에 포함:
   - 정상 케이스 (Happy Path)
   - 엣지 케이스 (PRD의 에러 시나리오 기반)
   - 경계값 (빈 값, 최대값, 음수, null)
3. 테스트 명세를 code-writer 프롬프트에 포함 — "이 테스트를 통과하게 구현할 것"
4. 구현 완료 후 code-writer가 테스트 코드도 작성

**테스트 없는 프로젝트**: 수동 검증 체크리스트로 대체

### Phase E: 리뷰 체크리스트 (PRD에 포함)

spec-reviewer와 quality-reviewer가 확인하는 **리뷰 체크리스트**를 PRD에 정의.

**공통 체크리스트** (모든 프로젝트):
- [ ] architect 인터페이스 계약서 준수 (시그니처, 타입 일치)
- [ ] 지시하지 않은 파일 수정하지 않았는가
- [ ] 에러 처리가 PRD의 에러 시나리오와 일치하는가
- [ ] 기존 코드 참조가 architect 지정 패턴과 동일한가
- [ ] 구현 제약사항(금지/강제)을 준수했는가
- [ ] 보안 이슈 없는가 (XSS, injection 등)

**프로젝트 특화** (PRD 작성 시 추가):
```markdown
- [ ] API 응답은 항상 { success, data, error } 형식
- [ ] Prisma 필드명이 스키마와 정확히 일치
- [ ] 인증이 필요한 모든 엔드포인트에 미들웨어 적용
```

### Phase F: 구현 (모든 준비 완료 후)
사용자에게 선택:
1. **일반 (1인)** — code-writer 1명이 순차 구현
2. **파이프라인 병렬** — 작업을 파일 단위로 분할, code-writer N명 병렬 (상세: `rules/subagent-rules.md` Pipeline Parallelization)

**code-writer 프롬프트에 포함할 것 (전부):**
- PRD (요구사항 + 에러 시나리오)
- 아키텍처 설계 (파일 구조, 인터페이스)
- 참고 구현체 (있으면)
- UI 디자인 참고 (프론트엔드면)
- 테스트 명세 (test-architect 설계)
- 리뷰 체크리스트

### Phase G: 검증 (코드 작성 완료 후)

4-Phase 피드백 루프 (Superpowers 2단계 리뷰 적용):

| Phase | 내용 | 최대 횟수 |
|-------|------|----------|
| 0. 셀프 리뷰 | code-writer가 자신이 작성한 코드를 Read로 재확인 후 자가 수정 | 1회 |
| 1. 스펙 준수 | spec-reviewer가 PRD 요구사항 일치 여부 검증 → 이슈 수정 → 재리뷰 | 2회 |
| 2. 코드 품질 | quality-reviewer가 보안/성능/패턴 검증 → 이슈 수정 → 재리뷰 | 2회 |
| 3. 빌드 + 테스트 | 빌드 → 테스트 → 실패 시 수정 | 3회 |
| **전체 합산** | | **5회** |

- **스펙 준수는 품질 리뷰보다 항상 먼저** — 순서 바꾸면 안 됨
- **Critical(>=90)**: 즉시 수정, **Important(80-89)**: 1회만 수정 시도
- **같은 에러 2회 연속**: 접근 방식 자체가 틀림 → 사용자 보고
- **5회 도달 시**: 즉시 중단, 상황 보고 + 다음 단계 제안
- 테스트 없는 프로젝트: Phase 3의 테스트만 스킵
- 코드 작성 없는 작업: 검증 스킵

### 리뷰 분할 전략 (대규모 프로젝트)

**10파일 이하**: spec-reviewer 1명 → quality-reviewer 1명 (순차)

**10파일+ (파이프라인 병렬 작업 시)**: 3단계 리뷰 적용

```
Phase 1 병렬 구현 직후 — Per-pipeline 2단계 리뷰 (병렬)
├── 파이프라인 A: spec-reviewer → quality-reviewer
├── 파이프라인 B: spec-reviewer → quality-reviewer
└── 파이프라인 C: spec-reviewer → quality-reviewer
    ↓ 이슈 수정

Phase 2 연결 완료 후 — 통합 리뷰 (1명씩 순차)
├── spec-reviewer — 파이프라인 간 스펙 준수, 인터페이스 일치
└── quality-reviewer — 전체 패턴, 보안, 성능 검증
    ↓ 이슈 수정

빌드 + 테스트
```

**Per-pipeline 리뷰 포커스:**
- spec-reviewer: 해당 파이프라인 PRD 요구사항 충족, 에러 시나리오
- quality-reviewer: 내부 로직 정확성, 코드 품질 (중복, 네이밍, 가독성)

**통합 리뷰 포커스:**
- spec-reviewer: 파이프라인 간 인터페이스 일치, 스펙 누락
- quality-reviewer: import 경로, 상태/데이터 흐름, 전체 아키텍처 준수
