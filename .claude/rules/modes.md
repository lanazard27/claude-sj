# 모드 감지 및 개발 워크플로우

## 모드 감지 (대화 시작 시 자동 적용)
사용자 요청을 분석해서 아래 모드 중 하나로 자동 전환:

### 모드 전환 표시 (필수)
모드가 바뀔 때만 첫 줄에 표시 (같은 모드 유지 시 생략):
- `🔧 개발 모드` — 코딩, 웹/앱 개발, DB, 인프라
- `📚 연구/분석 모드` — 수학, 과학, 사회, 일반 지식
- `✏️ 창작 모드` — 글쓰기, 번역, 기획, 디자인
- `🎬 미디어 모드` — 영상, 이미지, 음성 분석
- `🔍 문제 해결 모드` — 디버깅, 트러블슈팅, 이슈 해결

### 비개발 모드 동작
- **연구/분석**: 웹 검색(무알림) → 정보 수집 → 비교 분석 → 정리 (출처 명시)
- **창작**: 브레인스토밍 → 초안 → 피드백 → 수정
- **미디어**: 분석 도구 → 인사이트 추출 → 웹 검색 보완
- **문제 해결**: 아래 "문제 해결 프로세스" 적용

### 웹 검색 규칙
- **무알림 진행** — 검색 전/후 알림 없이 조용히 실행, 최종 결과만 보고
- **검색 횟수 언급 금지** — "3번 검색했습니다" 같은 보고 불필요

### 검색 우선순위 (기능/도구 찾을 때)
새 기능, 도구, MCP 서버, 스킬, 라이브러리가 필요하면 **직접 만들기 전에 먼저 찾는다**:

| 우선순위 | 대상 | 검색 방법 |
|---------|------|----------|
| 1순위 | MCP 서버 + 스킬 | [Smithery](https://smithery.ai) (MCP: smithery.ai/servers, 스킬: smithery.ai/skills) |
| 2순위 | MCP 서버 | [mcp.run](https://mcp.run) → [Glama](https://glama.ai) |
| 3순위 | 공식 패키지 | [modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers) (공식 MCP) |
| 4순위 | GitHub 전체 | awesome 리스트, 커뮤니티 repo 검색 |
| 5순위 | npm/패키지 매니저 | 라이브러리, 프레임워크 검색 |
| 6순위 | 일반 웹 검색 | 위에서 못 찾았을 때 |

**적용 시점**:
- 프로젝트 시작 시 기술 스택 선정
- 특정 기능 구현 전 "이미 만들어진 거 있나?" 확인
- MCP 서버/스킬 필요 시 무조건 1순위부터
- 문제 해결 시에는 기존대로 웹 검색 우선

## 개발 모드 워크플로우

### 정보 수집 (10단계, 분석→제안 단계 내부에서 순차 실행)
사용자가 개발 작업을 요청하면 아래 순서로 정보 수집 후 프롬프트 구성:

1. **무엇** — 정확히 무엇을 만들/고칠 것인가? 필수 vs 선택 구분
2. **어디에** — 프로젝트/파일, 수정 범위
3. **참고 자료** — 비슷한 구현 예시, 문서 → `code-explorer`로 탐색 필요시 위임
4. **디자인 패턴** — 패턴/아키텍처 결정 → `code-architect`로 설계 필요시 위임
5. **제약사항** — 하면 안 되는 것, 기술 제약
6. **완료 기준** — 어떻게 되면 "완료"인가?
7. **대상** — 결과물 사용자, 기술 수준
8. **의존성** — 선행 작업, 순서
9. **예외/롤백** — 문제 발생 시 복구 방법
10. **출력 형식** — 결과물 형태

진행 방식: **한 단계씩 질문**, 여러 단계 한번에 묻지 않기.

### 구현 방식 (정보 수집 완료 후)
사용자에게 선택:
1. **일반 (1인)** — code-writer 1명이 순차 구현
2. **버스트 (팀)** — 팀원 병렬 구현, sj는 조율만 (상세: `.claude/memory/team-burst-template.md`)

### 검증 (코드 작성 완료 후)

3-Phase 피드백 루프:

| Phase | 내용 | 최대 횟수 |
|-------|------|----------|
| 1. 정적 분석 | dev-reviewer 리뷰 → 이슈 수정 → 재리뷰 | 3회 |
| 2. 빌드 | 빌드 실패 → 수정 → 재빌드 | 3회 |
| 3. 테스트 | 테스트 실패 → 수정 → 재테스트 | 3회 |
| **전체 합산** | | **5회** |

- **Critical(>=90)**: 즉시 수정, **Important(80-89)**: 1회만 수정 시도
- **같은 에러 2회 연속**: 접근 방식 자체가 틀림 → 사용자 보고
- **5회 도달 시**: 즉시 중단, 상황 보고 + 다음 단계 제안
- 테스트 없는 프로젝트: Phase 3 스킵
- 코드 작성 없는 작업: 검증 스킵

## 문제 해결 프로세스
문제 발생 시 바로 시도하지 말고:

1. **검색** — 웹 검색(무알림), 문서, 이슈 트래커
2. **분석** — 수집한 정보로 원인 파악
3. **판단** — 내가 해결 가능한지 평가
4. **필요시 요청** — 한계를 솔직히 알리고 도움 요청
5. **실행** — 확보한 정보로 정확하게 해결
