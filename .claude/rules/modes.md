# 모드 감지 및 개발 워크플로우

## 모드 감지
사용자 요청 분석 후 자동 전환. 모드 변경 시 첫 줄에 표시:
- `🔧 개발` — 코딩, 웹/앱, DB, 인프라
- `📚 연구/분석` — 수학, 과학, 사회, 일반 지식. 웹 검색(무알림) → 수집 → 분석 → 정리
- `📚 Deep Research` — 복잡한 주제: 초기 쿼리 → 다중 홉 추론(최대 5회) → 품질 스코어링(0.6+만) → 종합 → 패턴 기록
- `✏️ 창작` — 글쓰기, 번역, 기획, 디자인. 브레인스토밍 → 초안 → 피드백 → 수정
- `🎬 미디어` — 영상, 이미지, 음성. 분석 도구 → 인사이트 추출 → 웹 검색 보완
- `🔍 문제 해결` — 디버깅, 트러블슈팅. 아래 "문제 해결 프로세스" 적용
- `🔥 핫픽스` — 긴급 수정, production 버그. 아래 "핫픽스 모드" 적용
- `🔌 스킬` — 스킬(데이터 수집) → sj 표준 파이프라인. 아래 "스킬 모드" 적용
- `📋 프로젝트 리딩` — 기존 프로젝트 문서 정리 + 에이전트 최적화 전달. 아래 "프로젝트 리딩 모드" 적용

**웹 검색**: 무알림 진행, 검색 횟수 언급 금지

### 검색 우선순위 (기능/도구 찾을 때)
1. **catalog-mcp** — MCP `search` (등록 시만, 응답 없으면 2순위부터)
2. **공식 마켓** — Claude Marketplaces, Skills Hub, npx skills find
3. **서드파티** — Smithery, mcp.run, Glama, modelcontextprotocol/servers, GitHub, npm
4. **일반 검색** — 위에서 못 찾았을 때

품질: 설치 수 1K+ 우선, 공식 출처 신뢰, GitHub 스타 <100 주의

## 프로젝트 리딩 모드

**진입**: "시작해" 시 프로젝트에 기존 문서(PRDO, ARCHITECTURE, MILESTONES, specs/ 등)가 있으면 sj가 자동 감지 → "문서가 이미 있네, 리딩 모드로 진행할까?" 질문 → 승인 시 진입
**역할**: sj가 문서 정리 + 에이전트 최적화 전달. 구현은 하지 않음.

### 워크플로우 — 마일스톤 브리프

1. **문서 파악** — 기존 문서 스캔 (PRD, 아키텍처, 스펙, UI 등)
2. **정합성 검증** — 문서 간 충돌/누락 확인 → 조정 제안
3. **마일스톤별 브리프 작성** — 아래 프로세스로 해당 마일스톤에 필요한 맥락만 발췌·구조화
4. **에이전트 전달** → 구현 진행

**브리프 프로세스**: 범위 파악(MILESTONES) → 문서별 발췌(아래 표) → 교차 참조 해결(에이전트가 직접 찾지 않게 연결) → 불필요 제거(다른 마일스톤/미래 Phase 제외) → 검증 포함(수용 기준 직접 삽입)

| 브리프 섹션 | 출처 | 발췌 내용 |
|------------|------|----------|
| 작업 범위 | MILESTONES + ARCHITECTURE | 생성/수정할 파일 경로 + 작업 유형 |
| 요구사항 | PRD + SPECS | 해당 마일스톤 기능 요구사항만 |
| 아키텍처 | ARCHITECTURE | 해당 패턴·디렉토리·인터페이스만 |
| 데이터 모델 | PRD + ARCHITECTURE | 관련 테이블/필드만 (전체 스키마 X) |
| 비즈니스 규칙 | BUSINESS_RULES | 적용 가능한 규칙만 |
| 에러 처리 | SPECS + API | 관련 에러 시나리오 + 에러코드 |
| UI 참고 | UI 목업 | 해당 화면만 |
| 수용 기준 | MILESTONES | 해당 마일스톤 검증 체크리스트 |
| 수정 금지 | — | 건드리면 안 되는 파일 목록 |

**전달 원칙**: 발췌 not 덤프 / 해결 not 참조("아키텍처 4.2"가 아니라 직접 삽입) / 표 형식 우선 / 경계 명확화(만들 것/안 만들 것) / 교차 정합성(다르게 설명되면 통일)

**구현 전환**: "개발 시작" → 개발 모드. Phase A~B는 기존 문서로 대체, Phase C~부터 진행.

## 개발 모드 워크플로우

### 규모 분기 (시작 시)
사용자가 "시작해" → sj가 프로젝트 디렉토리 확인:
1. **기존 문서 감지** — PRD/ARCHITECTURE/MILESTONES/specs/ 등 있으면 → "리딩 모드로 진행할까?" → 승인 시 `프로젝트 리딩 모드` 진입
2. **문서 없음** → 규모 질문:
| 규모 | 기준 | 워크플로우 |
|------|------|-----------|
| 소형 | 1-2파일, 단순 변경 | Phase A 미니 PRD → 바로 구현 |
| 중형 | 3-5파일, 신규 기능 | Phase A 스탠다드 PRD → Phase B~G |
| 대규모 | 5파일+, 다중 기능, 외부 연동 | 아래 "대규모 문서 제작" → Phase C~G |

### 대규모 문서 제작 (질문→정리→정합성)

**문서 제작 순서**:

| 순서 | 문서 | 내용 |
|------|------|------|
| 1 | PRD (PROJECT.md) | 프로젝트 정의, 사용자, 핵심 기능, 데이터 모델, 로드맵 |
| 2 | 비즈니스 규칙 (rules/BUSINESS_RULES.md) | 도메인 규칙, 검증 로직, 제약사항 |
| 3 | 아키텍처 (ARCHITECTURE.md) | 기술 스택, 레이어, 디렉토리, 패턴 |
| 4 | RBAC (rbac/RBAC.md) | 역할, 권한, 접근 제어 |
| 5 | 마일스톤 (MILESTONES.md) | 개발 단계, 의존관계, 검증 체크리스트 |
| 6 | 기능별 스펙 (specs/{domain}/SPEC.md) | 각 기능 상세 요구사항 |
| 7 | UI 목업 (ui/) | 화면 설계, 와이어프레임 |
| 8 | API/에러코드 (specs/api/) | 엔드포인트, 에러 정의 |

**각 문서 워크플로우**:
1. **질문** — 필요 항목을 3~4개씩 질문으로 끌어냄. 빈칸 없을 때까지 반복
2. **정리** — 답변을 문서 구조로 정리 → 사용자 확인 → 피드백 반영
3. **정합성** — 기존 문서와 충돌 확인. 변경 시 하위 문서 영향도 분석 → 조정 제안
4. **완료** → 다음 문서 진행

**정합성 관리**:
- **의존 맵**: `PRD → 비즈니스 규칙 → 아키텍처 → RBAC → 마일스톤 → 스펙 → UI → API`
- **변경 시**: 영향받는 하위 문서 표시 → 업데이트 → 사용자 확인
- **역추적**: 하위 문서 변경 시 PRD와 일치하는지 확인

모든 문서 완료 → Phase C~G 진행 (Phase A~B는 제작한 문서로 대체)

### 대규모 프로젝트 컨테이너 분할

**적용 조건**: 5파일+, 다중 기능, 복잡도 "중간" 이상

**분할 전략**: 1) C0(공통 기반) 먼저 — DB 스키마, 공유 타입, 공통 UI, 인터페이스 계약서, **공통 CSS 클래스명 목록**(병렬 writer 간 불일치 방지) 2) 기능별 컨테이너 순차 — 인증→핵심→부가(의존성 기반) 3) 한 번에 하나의 컨테이너만 작업 4) 인터페이스는 C0에서 미리 정의(Contract-First) 5) 통합은 모든 컨테이너 완료 후

**컨테이너 정의**: ID(C0,C1..) / 이름 / 범위(포함/제외) / 의존성(선행 ID) / 산출물(파일 목록) / 인터페이스(타입, API, 이벤트)

**작업 프로세스**: 1) sj가 컨테이너 분할 제시 2) 사용자와 범위/순서 합의 3) 전체 PRD 선작성(PRD-C0.md ~ PRD-CN.md, 인터페이스/의존성 확정, 내부 디테일은 구현 시 보완) 4) 순차 구현 — 각 컨테이너 PRD 읽고 바로 architect→writer→reviewer 5) 완료 후 code-integrator 통합 + 전체 테스트

**인터페이스 계약 원칙**: Barrel Export(index.ts에서 공개 API만) / 최소 지식(인터페이스만 참조) / Stub/Mock 가능 / 파일 겹침 금지(C0는 예외) / 의존성 단방향(하위 참조, 상호 참조 금지)

### Phase A: 대화 → PRD 초안

**원칙**: 정보 수집과 PRD 작성을 분리하지 않는다.

#### Step 0: Confidence Check
| 신뢰도 | 판단 | sj 행동 |
|--------|------|---------|
| ≥90% | 명확함 | 바로 Step 1 → PRD 작성 |
| 70-89% | 애매함 | Step 1 + 추가 질문 보완 |
| <70% | 불충분 | 구체적 정보 요청, PRD 보류 |

#### Step 0.5: 카탈로그 사전 제안 (빠른 체크)
catalog-mcp `search`로 키워드 빠른 조회 → 관련 도구 있으면 제안. MCP 응답 없으면 스킵.

#### Step 1: 핵심 파악
**무엇을**(정확히 뭘 만들/고칠 거야?) / **어디에**(어떤 프로젝트/파일?) / **왜**(필요한 이유?)

#### Step 2: PRD 초안 (복잡도에 따라 3단계)
| 등급 | 판단 기준 | 워크플로우 |
|------|----------|-----------|
| 미니 PRD | 1-2파일, 기존 패턴, 버그 수정/간단 변경 | PRD 스킵, 대화로 핵심만 확인 후 바로 구현 |
| 스탠다드 PRD | 3-5파일, 신규 기능 | 아래 축소 구조 사용 |
| 풀 PRD | 5파일+, 다중 기능, 외부 연동 | `/prd` 스크로 참조 |

**스탠다드 PRD 구조**:
```markdown
# PRD: [기능명]
## 개요: [배경 + 목표 + 대상]
## 기능 요구사항: | ID | 요구사항 | 필수/선택 |
## 에러 시나리오: | 시나리오 | 예상 동작 |
## 수정 범위: | 파일 | 작업 | 설명 |
## 수용 기준: - [ ] [구체적 조건]
## UI/디자인 토큰 (스킬 모드 전용, 비스킬 모드는 제외)
### 색상 매핑: | 역할 | 색상값 | 용도 |
### 타이포그래피: | 요소 | 폰트 | 크기 | 굵기 |
### 레이아웃 수치: | 요소 | 크기 | 위치 |
### 인터랙션: | 애니메이션 티어 | 감지된 라이브러리 | 전환 속도 |
```

#### Step 3: PRD 디테일 보완
빈칸/애매한 부분만 질문: 에러 시나리오 비어있음 → 수용 기준 모호함 → 엣지 케이스 부족함 → 기술 스펙/아키텍처/API/데이터 모델/화면 흐름/외부 연동/수정 범위 중 빈 것.
**질문 원칙**: 섹션별 부족한 것만, 답변된 것은 채워넣고, 최대 3~4개, 한 번에 한 질문.

#### Step 4: PRD 확정
보완된 PRD를 사용자에게 보여줌 → 피드백 반영 → 확정.
**PRD 위치**: `/Users/jeongsunjae/projects/{프로젝트명}/PRD-{C0..CN}.md` (컨테이너별 개별 파일, 단일 시 `PRD.md`). **미니 PRD**: 파일 생성 안 함.

#### Step 4.5: 카탈로그 정밀 매칭 (PRD 확정 후)
PRD 전체 텍스트를 catalog-mcp `match_prd`에 전달 → 유용한 도구 발견 시 "이거 설치하면 구현 생략 가능" 제안. 승인 시 설치 후 Phase B. MCP 응답 없으면 스킵.
**Step 0.5 vs 4.5**: 0.5는 PRD 전 `search`(키워드, 빠름), 4.5는 PRD 후 `match_prd`(전체 분석, 정밀).

#### PRD 변경 프로세스 (구현 중 변경 시)
1. 영향도 분석(이미 구현된 부분인지 판단) 2. 미구현→PRD 업데이트 후 기존 파이프라인 계속 3. 이미 구현됨→architect에 재설계 필요 여부 질문 → 재설계 필요면 Phase B부터 재시작(기존 구현 보존) / 수정만 필요면 writer에게 변경 사항만 전달

### Phase B: 탐색 + 설계

**조건부 라우팅**: 기존 프로젝트(1개+ 파일)→code-explorer→architect. 신규(0개 파일)→explorer 스킵→architect가 PRD만으로 설계(표준 패턴 사용, 참고 구현체 필요시 웹 검색 보완).
**공통**: architect 완료 후 자동 검증(subagent-rules.md 참조), 필요 환경변수 포함(변수명/용도/필수여부/.env.example), 새 의존성 감지 시 sj가 Phase F 전에 설치(패키지/버전/용도)

### Phase C: UI 디자인 참고 (프론트엔드 시)
1. 레퍼런스 사이트 — URL 제공받으면 web-reader로 스타일 추출
2. Excalidraw 목업 — `/excalidraw-diagram-generator` 스킬로 와이어프레임
3. 디자인 토큰 — 색상, 폰트, 간격을 PRD에 명시 (없으면 Tailwind 기본값)

### Phase D: 테스트 명세 (구현 전, TDD)
1. test-architect가 PRD+아키텍처 기반 테스트 케이스 설계 2. 정상/엣지/경계값 명세 3. 명세를 writer 프롬프트에 포함 4. writer가 테스트 코드도 작성. **테스트 없는 프로젝트**: 수동 검증 체크리스트로 대체

### Phase E: 리뷰 체크리스트 (PRD에 포함)
- [ ] architect 인터페이스 계약서 준수
- [ ] 지시하지 않은 파일 수정하지 않았는가
- [ ] 에러 처리가 PRD 에러 시나리오와 일치
- [ ] 기존 코드 참조가 architect 지정 패턴과 동일
- [ ] 구현 제약사항(금지/강제) 준수
- [ ] 보안 이슈 없음 (XSS, injection 등)

### Phase F: 구현
1. **일반(1인)** — code-writer 1명 순차 구현
2. **파이프라인 병렬** — 파일 단위 분할, code-writer N명 병렬 (상세: `subagent-rules.md`)
**writer 프롬프트 포함**: PRD, 아키텍처 설계, 참고 구현체, UI 디자인 참고, 테스트 명세, 리뷰 체크리스트

### Phase G: 검증

4-Phase 피드백 루프:
| Phase | 내용 | 최대 |
|-------|------|------|
| 0. 셀프 리뷰 | writer가 Read로 재확인 후 자가 수정 | 1회 |
| 1. 스펙 준수 | spec-reviewer가 PRD 일치 검증 → 수정 → 재리뷰 | 2회 |
| 2. 코드 품질 | quality-reviewer가 보안/성능/패턴 검증 → 수정 → 재리뷰 | 2회 |
